==================== FASE 8 — PUBLICAR (solo si la FASE 7 fue 100% verde) ====================
Candados de publicación (TODOS deben cumplirse): el verificador dio ok=true; el diff toca como máximo 18 archivos
de PÁGINA HTML (corrida normal: 1–8; un diff mayor en un sitio de ~100 páginas es señal de que algo se salió de
alcance → lo ve un humano). NO cuentan contra ese cap: un cambio de ASSET compartido (styles*.css + el bump de ?v=
en las páginas que lo referencian + sw.js — UN cambio lógico aunque toque muchas) ni los borrados de artefactos
huérfanos que pasaron limpiar-huerfanos.py. Sí se prohíben los borrados ESTRUCTURALES inesperados (una página
indexable, una entrada de sitemap, un asset en uso). Si cualquiera falla: NO publiques; deja la rama y explica en
ESTADO.md. Si pasan TODOS, publica sincronizando ANTES con el remoto (git push --force PROHIBIDO):
  a. `git checkout main`
  b. `git fetch origin` y `git merge --ff-only origin/main`. Si el --ff-only FALLA (la main local divergió):
     ABORTA, deja la rama sin fusionar, escribe en ESTADO.md "publicación detenida: main local divergió" y termina.
  c. `git merge --no-ff <rama> -m "auto-diario $(date +%Y-%m-%d): N arreglos + M páginas"`
  d. `git push` (el pre-push auto-indexa en GSC; deja que corra).
  e. Si el push es RECHAZADO: NO uses --force NI rebase (rebase aplanaría el merge --no-ff y dejaría la rama
     "sin fusionar" → el branch -d fallaría y la rama quedaría colgada). Reintegra UNA vez con re-merge LIMPIO:
     `git fetch origin && git reset --hard origin/main && git merge --no-ff <rama> -m "<mismo mensaje>" && git push`.
     Si ese segundo push también falla, ABORTA, deja la rama, anota el motivo en ESTADO.md y termina.
  f. Solo si el push tuvo éxito: borra la rama fusionada. Luego, para páginas nuevas indexables, refuerza la
     indexación por MCP: mcp__gsc__gsc_index(site, [urls nuevas]).

==================== FASE 9 — APRENDER (agente que hace el sistema más inteligente) ====================
Lanza un subagente APRENDIZ (Task) cuyo único objetivo es que el MISMO error NO se repita y se detecte MECÁNICAMENTE
la próxima vez. Para cada error corregido este día —sobre todo REGRESIONES y errores humanos de contenido—:
  a. Añade o CONSOLIDA (no dupliques) una regla de PREVENCIÓN en REGLAS.md: qué pasó, por qué, y la regla dura para
     no repetirlo. Si varias reglas se solapan, fúndelas.
  b. Si el error es mecanizable, propón el CHECK que lo cazaría solo y, si es de bajo riesgo, AÑÁDELO a
     .pipeline/check-plantilla.py (o el checker que corresponda) para que de ahora en más sea un hallazgo
     determinista y deje de depender de que un LLM "se acuerde". Verifica que el checker sigue imprimiendo JSON
     válido tras tu cambio (córrelo).
  c. Registra en HISTORIAL.jsonl una línea por error (fecha, archivo, categoria, severidad, descripcion, estado
     "arreglado"/"pendiente", regla aplicada/creada). El sistema se vuelve más listo midiéndose por esto: cada
     clase de error vista una vez debe quedar blindada por una regla o un checker.
