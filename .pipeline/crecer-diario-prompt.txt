==================== FASE 7 — VERIFICAR (agente independiente: "¿quedó todo bien?") ====================
Lanza un subagente VERIFICADOR (Task, general-purpose) con instrucción de ser ESCÉPTICO: su trabajo es intentar
demostrar que algo quedó MAL, no confirmar que está bien. Debe, sobre la rama de la corrida:
  a. Releer `git diff main --stat` y confirmar que el alcance es razonable y nada quedó fuera de lo previsto.
  b. Re-correr los checkers deterministas desde cero: `python3 .pipeline/ci-gate.py` (0 ALTA) y, por CADA página
     nueva/modificada, `python3 .pipeline/gate-pagina.py <ruta/index.html>` (validate + checkers + anti-doorway).
  c. Con el sitio corriendo, cargar cada página tocada y confirmar HTTP 200, JSON-LD que parsea,
     canonical==og:url==twitter:url, 0 enlaces internos rotos, y que cada arreglo de la FASE 5 realmente se aplicó.
  d. Confirmar que NO se tocaron precios, tests, ni se borraron páginas; que el email no quedó contaminado; que
     ninguna página nueva quedó huérfana (tiene enlace entrante).
El verificador entrega un veredicto JSON {ok: true/false, problemas: [...]}. Si ok=false o encuentra CUALQUIER
problema: NO se publica nada; deja la rama, escribe el motivo en ESTADO.md y salta a la FASE 9 (aprender) y email.

==================== FASE 8 — PUBLICAR (solo si la FASE 7 fue 100% verde) ====================
Candados de publicación (TODOS deben cumplirse): el verificador dio ok=true; el diff toca como máximo 18 archivos
(corrida normal: 1–8; un diff mayor en un sitio de ~100 páginas es señal de que algo se salió de alcance → lo ve
un humano); sin borrados estructurales inesperados. Si cualquiera falla: NO publiques; deja la rama y explica en
ESTADO.md. Si pasan TODOS, publica sincronizando ANTES con el remoto (git push --force PROHIBIDO):
  a. `git checkout main`
  b. `git fetch origin` y `git merge --ff-only origin/main`. Si el --ff-only FALLA (la main local divergió):
     ABORTA, deja la rama sin fusionar, escribe en ESTADO.md "publicación detenida: main local divergió" y termina.
  c. `git merge --no-ff <rama> -m "auto-diario $(date +%Y-%m-%d): N arreglos + M páginas"`
  d. `git push` (el pre-push auto-indexa en GSC; deja que corra).
  e. Si el push es RECHAZADO: NO uses --force. Reintegra UNA vez (`git fetch origin && git rebase origin/main &&
     git push`); si el segundo push también falla, ABORTA, deja la rama, anota el motivo en ESTADO.md y termina.
  f. Solo si el push tuvo éxito: borra la rama fusionada. Luego, para páginas nuevas indexables, refuerza la
     indexación por MCP: mcp__gsc__gsc_index(site, [urls nuevas]).

  b. Si el error es mecanizable, propón el CHECK que lo cazaría solo y, si es de bajo riesgo, AÑÁDELO a

     determinista y deje de depender de que un LLM "se acuerde". Verifica que el checker sigue imprimiendo JSON

  c. Registra en HISTORIAL.jsonl una línea por error (fecha, archivo, categoria, severidad, descripcion, estado
